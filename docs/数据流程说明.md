## 二、第一步：表单输入与前端校验

### 2.1 Vue 模板绑定

```html
<!-- RegisterView.vue:6-9 -->
<input type="text" v-model="username" placeholder="用户名（至少2个字符）" />
<input type="password" v-model="password" placeholder="密码（至少6个字符）" />
<input type="password" v-model="confirmPassword" placeholder="确认密码" />
<button type="submit" :disabled="submitting">{{ submitting ? '注册中...' : '注册' }}</button>
```

**设计意图**：

- **`v-model` 双向绑定**：用户的每次输入都在 Vue 响应式系统内部更新 `username`、`password`、`confirmPassword` 三个 ref。这不是简单的 setter——Vue 3 用 Proxy 代理这些值，任何修改都会触发依赖追踪，确保模板重新渲染。
- **`:disabled="submitting"`**：这是防重复提交的第一道防线。当点击按钮后 `submitting` 立即可见地设为 `true`，即使网络请求卡了几秒，按钮也不会响应第二次点击。
- **`<button type="submit">`**：触发 `<form @submit.prevent="handleRegister">`，`.prevent` 阻止浏览器默认行为（刷新页面）。

### 2.2 JavaScript 校验逻辑

```js
// RegisterView.vue:31-47
const submitting = ref(false)
const error = ref('')

async function handleRegister() {
  // ── 第一道闸：防重复提交 ──
  if (submitting.value) return             // 已提交中，直接忽略

  // ── 第二道闸：密码一致性 ──
  if (password.value !== confirmPassword.value) {
    error.value = '两次输入的密码不一致'
    return
  }

  // ── 锁定按钮，进入提交状态 ──
  submitting.value = true
  error.value = ''

  try {
    await authStore.register(username.value, password.value, confirmPassword.value)
    router.push('/')                        // 成功后跳转主页
  } catch (e) {
    error.value = e.response?.data?.detail || '注册失败'
    // ⚠ 后端返回的 Pydantic ValidationError 结构：
    //   e.response.data.detail = [{loc: ["body","username"], msg: "..."}]
    //   这里取了第一个 detail，生产环境可解析所有字段错误
  } finally {
    submitting.value = false               // 无论成败都解锁按钮
  }
}
```

**关键考量**：

- `submitting` 在 `try` 之前就设为 `true`——如果放在 `try` 里第一行，理论上存在竞态窗口。
- `e.response?.data?.detail` 使用可选链 `?.`，因为网络断开时 `e.response` 可能是 `undefined`（axios 自己抛出的网络错误没有 `response` 属性）。

### 2.3 数据形态变化

| 阶段 | 数据形态 |
|------|----------|
| 用户输入 | `{username: "testuser", password: "test1234", confirm: "test1234"}` — 模板绑定 |
| 校验通过 | 同上，但 `submitting` 已切换到 `true`，按钮 disabled |
| 进入 Store | 作为参数传给 `authStore.register()` |

## 三、第二步：Pinia Store 发起注册

### 3.1 Store Action

```js
// stores/auth.js:21-26
async function register(username, password, confirmPassword) {
  // ↓ 调用 API 层，数据到这一步仍是明文
  const res = await apiRegister({
    username,
    password,                     // 明文，尚未哈希
    confirm_password: confirmPassword  // 蛇形命名，与后端 Pydantic schema 对齐
  })

  // ↓ 后端返回后，立即持久化到状态和浏览器
  token.value = res.data.access_token    // Pinia ref — Vue 响应式状态
  user.value = res.data.user             // Pinia ref — 触发 NavBar 等组件更新
  localStorage.setItem('token', token.value)  // 浏览器持久化 — 刷新不丢失
}
```

**设计意图**：

- **Store 负责状态管理**：`token` 和 `user` 是 Pinia 的 ref。一旦赋值，所有依赖它的组件（`NavBar.vue` 显示用户名/管理员徽章、`HomeView.vue` 的 `isAdmin` computed）都会自动重新渲染。
- **localStorage 持久化**：为什么不用 Cookie？因为前端是 SPA 单页应用，axios 拦截器直接读 localStorage 注入 Authorization 头比设置 httpOnly Cookie 简单。代价是如果未来引入 XSS，token 可直接被窃取。**这是已知的技术债务**。
- **`confirm_password` 蛇形命名**：Python/Pydantic 默认蛇形 `confirm_password`，JavaScript 默认驼峰 `confirmPassword`。这里的参数名故意写为蛇形，因为 `api/auth.js` 直接 `api.post('/auth/register', data)` 传入，axios 会 JSON.stringify 后发送，后端 FastAPI 期望收到的是蛇形键名。

### 3.2 API 封装层

```js
// api/auth.js:5 — 一行封装，隔离路由路径
export const register = (data) => api.post('/auth/register', data)
```

**设计意图**：

这是一个"浅模块"。它的唯一价值是把 URL 路径 `/auth/register` 与组件解耦——如果后端改为 `/api/v2/auth/register`，只需修改这一个文件。但有些人会认为这过于间接，store 可以直接 `api.post('/auth/register', data)`。两种观点都有道理，这里选择了显式封装。

## 四、第三步：Axios 拦截器链

### 4.1 请求拦截器

```js
// api/index.js:17-22
api.interceptors.request.use((config) => {
  const token = localStorage.getItem('token')
  if (token) {
    config.headers.Authorization = `Bearer ${token}`
  }
  return config
})

// 发出去的请求头：
// POST /api/auth/register HTTP/1.1
// Host: localhost:5173
// Content-Type: application/json       ← axios 自动设置
// Authorization: (无，首次注册无 token)
//
// { "username":"testuser","password":"test1234","confirm_password":"test1234" }
```

**设计意图**：

- **拦截器集中管理**：每个请求自动注入 token，不用在每个 API 调用中手动传 headers。这叫"横切关注点"（cross-cutting concern）。
- **首次注册**：此时 `localStorage.getItem('token')` 返回 `null`，拦截器跳过注入。这是正确行为——注册不需要认证。
- **生产环境的隐含问题**：密码在 HTTP body 中以明文 JSON 传输。开发环境 localhost 无所谓，生产环境必须有 HTTPS 才能确保中间人无法窃听。这是传输层的问题，不是代码层的问题。

### 4.2 响应拦截器

```js
// api/index.js:26-35
let isRedirecting = false    // 模块级变量，跨请求共享

api.interceptors.response.use(
  (res) => res,              // 2xx → 直接放行

  (err) => {
    // 只有 401 且未在重定向中才处理
    if (err.response?.status === 401 && !isRedirecting) {
      isRedirecting = true              // 设置标志，防并发 401 多次跳转
      localStorage.removeItem('token')   // 清除过期 token
      window.location.href = '/login'    // 硬刷新跳转，清空 Vue 状态
    }
    return Promise.reject(err)           // 继续向上抛出，让调用方 catch 处理
  }
)
```

**关键考量**：

- **`isRedirecting` 标志**：这是一个模块级（非 ref）变量。如果不用这个标志，当页面有多个并发请求同时返回 401（如 token 过期时 HomeView 同时发起 markers + maps + regions 三个请求），会触发三次 `window.location.href = '/login'`，导致浏览器历史记录混乱。
- **不可用 `router.push`**：`api/index.js` 是纯 JS 模块，在 Vue 插件链的最底层被 import。此时 Vue Router 还未初始化（循环依赖），所以只能使用 `window.location.href` 硬跳转。这是一个已知的架构约束。
- **`localStorage.removeItem`** 必须在跳转前执行，否则跳转后 `router.beforeEach` 守卫会读到残留的 token 认为用户已登录。

### 5.2 代理做了什么

```
浏览器端 Vue 应用                        Vite Dev Server                   FastAPI
     │                                       │                                │
     │  POST /api/auth/register              │                                │
     │  Host: localhost:5173                 │                                │
     ├──────────────────────────────────────→│                                │
     │                                       │  路径 /api 开头，触发代理规则    │
     │                                       │  Host 改为 127.0.0.1:8000       │
     │                                       ├───────────────────────────────→│
     │                                       │                                │
     │                                       │←───────────────────────────────┤
     │←──────────────────────────────────────│                                │
     │                                       │                                │
```

**设计意图**：

- **同源策略绕过**：浏览器禁止 `localhost:5173` 的页面向 `localhost:8000` 发送跨域请求（不同端口即不同源）。代理让浏览器认为请求和响应来自同一个源（5173），实际上请求被转发到 8000。
- **生产环境**：部署时由 Nginx 替代 Vite Proxy，配置类似 `location /api { proxy_pass http://backend:8000; }`。
- **`changeOrigin: true`**：将请求头中的 `Host` 从 `localhost:5173` 改为 `127.0.0.1:8000`。否则后端可能根据 Host 头做路由判断，导致错误。

### 6.2 依赖注入链

FastAPI 收到 POST `/api/auth/register` 后，按以下顺序解析依赖：

```
1. rate_limit_auth(request)
   ├─ 提取客户端 IP
   ├─ 检查频率计数
   ├─ 超限 → 抛出 HTTPException(429)
   └─ 未超限 → 放行

2. get_db()
   ├─ db = SessionLocal()          ← 创建数据库会话
   ├─ yield db                     ← 注入到 register 函数
   └─ 请求结束后 db.close()        ← finally 块自动关闭

3. register(data, db, _)
   ├─ data: fastapi 自动将请求 body JSON 反序列化为 UserRegister Pydantic 对象
   ├─ db:   get_db 返回的 SQLAlchemy Session
   └─ _:     rate_limit_auth 的返回值（未使用，用下划线忽略）
```

**关键细节**：

- **`Depends` 可以链式嵌套**：`require_admin = Depends(get_current_user)` 在依赖中又依赖另一个依赖。FastAPI 的依赖解析器可以处理任意深度的嵌套。
- **`yield` 而非 `return`**：`get_db` 用 `yield` 而非 `return` 是为了在请求结束后执行 `finally` 块中的 `db.close()`。这是 Python 生成器的上下文管理特性。
- **`response_model=TokenResponse`**：FastAPI 会在返回前自动将返回值序列化为 `TokenResponse` 模型，并执行字段校验和 JSON 序列化。如果返回值不符合模型定义，返回 500 给客户端。

## 七、第六步：频率限制拦截

### 7.1 RateLimiter 核心实现

```python
# rate_limit.py:13-41
class RateLimiter:
    def __init__(self, max_requests: int = 5, window_seconds: int = 60):
        self.max_requests = max_requests           # 窗口内最大请求数
        self.window_seconds = window_seconds       # 滑动窗口大小（秒）
        self._store = defaultdict(list)             # key → [timestamp, ...]
        self._hits = 0                              # 全局命中计数（用于触发清理）

    def _clean(self, key: str, now: float):
        """清理单个 key 的过期时间戳"""
        cutoff = now - self.window_seconds
        self._store[key] = [t for t in self._store[key] if t > cutoff]
        if not self._store[key]:
            del self._store[key]

    def _full_clean(self, now: float):
        """每 1000 次调用全量扫描一次，清理所有过期 key"""
        cutoff = now - self.window_seconds
        expired = [k for k, v in self._store.items() if not any(t > cutoff for t in v)]
        for k in expired:
            del self._store[k]

    def is_allowed(self, key: str) -> bool:
        now = time.time()
        self._clean(key, now)
        self._hits += 1
        if self._hits % 1000 == 0:          # 每 1000 次做一次全量清理
            self._full_clean(now)
        if len(self._store[key]) >= self.max_requests:
            return False                      # 超限
        self._store[key].append(now)          # 记录本次时间戳
        return True                           # 放行
```

### 7.2 应用到注册路由

```python
# rate_limit.py:44
_auth_limiter = RateLimiter(max_requests=10, window_seconds=60)
# 注册和登录两个端点共用的限流器：10次/分钟

# rate_limit.py:47-57
def rate_limit_auth(request: Request):
    forwarded = request.headers.get("X-Forwarded-For")
    if forwarded:
        client_ip = forwarded.split(",")[0].strip()   # 反向代理后取真实 IP
    else:
        client_ip = request.client.host if request.client else "unknown"
    key = f"auth:{client_ip}"
    if not _auth_limiter.is_allowed(key):
        raise HTTPException(status_code=429, detail="请求过于频繁，请稍后再试")
```

**设计意图**：

- **滑动窗口而非固定窗口**：每个请求的时间戳都记录下来，过期的自动清理。这比"每整分钟重置计数器"更平滑，避免恰好在窗口边界被限流。
- **X-Forwarded-For 处理**：部署在 Nginx 后面时，`request.client.host` 永远是 Nginx 的 IP（如 `172.17.0.1`）。必须读 `X-Forwarded-For` 头取真实客户端 IP。
- **登录和注册共享限流器**：这是有意的设计——攻击者可以用同一个 IP 交替尝试登录和注册来绕过限流，共享计数器堵住了这个口子。

### 7.3 已知局限

- **多 worker 不共享**：`_store` 是进程内存字典。uvicorn 开 4 个 worker 时有 4 份独立计数器，攻击者可以在 4 个 worker 各打 9 次。生产需 Redis。
- **`_full_clean` 只清理全过期的 key**：一个 key 只要有一个在窗口内的时间戳，就不会被删除。高频 IP 的 `_store[key]` 列表会不断增长直到被清理。这是 O(n) 内存使用。

## 八、第七步：Pydantic 参数校验

### 8.1 UserRegister Schema

```python
# schemas.py:13-41
class UserRegister(BaseModel):
    username: str
    password: str
    confirm_password: Optional[str] = None

    @field_validator("username")
    @classmethod
    def username_valid(cls, v: str) -> str:
        v = v.strip()
        if len(v) < 2:
            raise ValueError("用户名至少需要2个字符")
        if len(v) > 50:
            raise ValueError("用户名不能超过50个字符")
        if not re.match(r'^[\w\u4e00-\u9fff]+$', v):
            raise ValueError("用户名只允许字母、数字、下划线和中文")
        return v

    @field_validator("password")
    @classmethod
    def password_valid(cls, v: str) -> str:
        if len(v) < 6:
            raise ValueError("密码至少需要6个字符")
        return v

    @field_validator("confirm_password")
    @classmethod
    def passwords_match(cls, v: Optional[str], info) -> Optional[str]:
        if v is not None and "password" in info.data and v != info.data["password"]:
            raise ValueError("两次输入的密码不一致")
        return v
```

### 8.2 校验执行顺序

FastAPI 收到请求 body `{"username":"testuser","password":"test1234","confirm_password":"test1234"}` 后：

```
1. JSON 反序列化
   {"username":"testuser",...} → Python dict

2. Pydantic BaseModel.__init__
   按字段定义顺序初始化：
   ├─ username = "testuser"
   │   └─ @field_validator("username")  → strip → 长度检查 → 正则检查
   │
   ├─ password = "test1234"
   │   └─ @field_validator("password")  → 长度检查
   │
   └─ confirm_password = "test1234"
       └─ @field_validator("confirm_password")  → 与 password 比较

3. 校验通过 → 返回 UserRegister 对象
4. 校验失败 → 抛出 ValidationError → FastAPI 自动转为 422 响应
```

### 8.3 正则表达式解析

```python
re.match(r'^[\w\u4e00-\u9fff]+$', v)
```

| 部分 | 含义 |
|------|------|
| `^` | 字符串开头 |

## 九、第八步：数据库查重 + 写入 + 密码哈希

### 9.1 查重逻辑

```python
# routers/auth.py:19-21
existing = db.query(User).filter(User.username == data.username).first()
if existing:
    raise HTTPException(status_code=400, detail="用户名已存在")
```

**双重保障**：

1. **Python 层**：SQLAlchemy 查询后手动判断。即使数据库唯一约束不生效（某些云数据库跳过索引），这里也能拦住。
2. **数据库层**：`username VARCHAR(50) UNIQUE NOT NULL` 约束。即使 Python 层被绕过（比如多个 worker 同时插入），数据库也会报 `Duplicate entry` 错误。

### 9.2 密码哈希

```python
# auth.py:18 — 全局单例
pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

# auth.py:23-25
def hash_password(password: str) -> str:
    return pwd_context.hash(password)
```

**bcrypt 原理**：

```
输入: "test1234"
  ↓
1. 生成随机 16 字节盐 (salt)
2. 将密码 + 盐输入 Blowfish 加密算法，迭代 2^12 = 4096 次
3. 输出 $2b$12$[22字符salt][31字符哈希]

示例: $2b$12$LJ3m4ys3GZ5k7Q2v8xWnFuPf9RxVYhMBd1TUKlzONpAqWjHeSrCKe
       ──┬── ─┬─ ───────────┬─────────── ──────────────┬──────────────
      版本  cost   22字符 salt                      31字符哈希
```

**关键属性**：

- **不可逆**：从哈希反推明文在数学上不可行（Blowfish 是单向函数）
- **加盐**：即使两个用户密码相同，由于盐不同，哈希完全不同
- **抗彩虹表**：彩虹表无法预计算所有可能的盐值，需要为每个 salt 单独建表
- **cost=12**：迭代 2^12 次，约 0.1~0.3 秒，足够慢以防止暴力破解

### 9.3 ORM 写入

```python

    username=data.username,
    password_hash=hash_password(data.password),
```

**Session 机制**：

```

   ↓ 此时数据仅在 Python 对象中，未发送 SQL
db.commit()
   ↓
   1. session flush: 生成 INSERT INTO users (...) VALUES (...)
   2. 发送到 MySQL
   3. MySQL 执行 INSERT，分配自增 id=5
   4. COMMIT 事务
db.refresh(user)
   ↓
   发出 SELECT * FROM users WHERE id=5
   获取数据库层面的默认值（如 created_at 由 MySQL NOW() 设置）
   user.id 现在等于 5
```

## 十、第九步：JWT 令牌签发

### 10.1 签发流程

```python
# auth.py:33-38
def create_access_token(data: dict) -> str:
    to_encode = data.copy()                            # {"user_id": 5}
    expire = datetime.now(timezone.utc) + timedelta(minutes=JWT_EXPIRE_MINUTES)
    to_encode.update({"exp": expire})                  # 添加过期时间
    return jwt.encode(to_encode, JWT_SECRET_KEY, algorithm=JWT_ALGORITHM)
```

### 10.2 JWT 结构拆解

JWT 由三部分组成，用 `.` 分隔：

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9           ← Header (Base64 编码)
.eyJ1c2VyX2lkIjo1LCJleHAiOjE3ODA5NjczMjZ9       ← Payload (Base64 编码)
.r__jbCE_2Htcp4zLvB55dKxvsv8aU2KA3fJOqoEzEoo    ← Signature (HMAC-SHA256)
```

| 部分 | Base64 解码后 |
|------|--------------|
| Header | `{"alg":"HS256","typ":"JWT"}` |
| Payload | `{"user_id":5,"exp":1780967326}` |
| Signature | `HMAC-SHA256(Header.Payload, JWT_SECRET_KEY)` |

### 10.3 签名的作用

```
伪造 token 的真实困难：
  攻击者知道 Header + Payload 是明文 Base64
  攻击者可以随意修改 Payload 中的 user_id 从 5 改为 1 (admin)
  但攻击者无法生成有效的 Signature
    — 因为 Signature = HMAC-SHA256(内容, 密钥)
    — 密钥只存在于服务端 .env 文件的 JWT_SECRET_KEY 中

验证流程：
  jwt.decode(token, JWT_SECRET_KEY, algorithms=["HS256"])
  1. 用密钥重新计算 HMAC-SHA256(Header.Payload)
  2. 与 token 中的 Signature 比较
  3. 不一致 → JWTError → 401 Unauthorized
```

### 10.4 配置安全

```python
# config.py:21-24 — 启动时强制验证
JWT_SECRET_KEY = os.getenv("JWT_SECRET_KEY")
if not JWT_SECRET_KEY:
    print("错误: 未设置 JWT_SECRET_KEY 环境变量", file=sys.stderr)
    sys.exit(1)     # 应用启动失败，拒绝在无密钥状态下运行

# config.py:27-31 — 算法白名单，防止 "none" 算法攻击
_ALLOWED_ALGORITHMS = frozenset({"HS256", "HS384", "HS512", "RS256", "RS384", "RS512"})
if _raw_algo not in _ALLOWED_ALGORITHMS:
    sys.exit(1)
```

**为什么需要算法白名单**：JWT 规范支持 `"alg":"none"`（无签名）。如果攻击者篡改 `.env` 设置 `JWT_ALGORITHM=none`，任何伪造的 token 都会被接受。白名单彻底杜绝了这种可能。

## 十二、第十一步：登录态恢复

### 12.1 App.vue 启动逻辑

```js
// App.vue:27-29
onMounted(() => {
  authStore.fetchUser()
})
```

`App.vue` 是 Vue 应用的根组件，`onMounted` 在首次渲染后执行。`fetchUser()` 在两种场景下都会执行：

- **场景 A — 刚注册完**：token 已存，`getMe()` 返回用户信息，确认 token 有效
- **场景 B — 页面刷新**：localStorage 中有上次的 token，用 `getMe()` 验证是否过期

### 12.2 fetchUser 实现

```js
// stores/auth.js:28-40
async function fetchUser() {
  if (!token.value) return               // 无 token → 跳过

  try {
    const res = await getMe()            // GET /api/auth/me
    user.value = res.data
  } catch (e) {
    if (e.response?.status === 401) {
      logout()                           // token 过期 → 清除状态
    } else {
      console.warn('fetchUser 非 401 错误:', e.response?.status)
      // 网络错误/服务器 500 → 不登出，容忍短暂故障
    }
  }
}
```

### 12.3 getMe 后端的验证链

```
GET /api/auth/me
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

→ routers/auth.py:49-52
  def get_me(current_user=Depends(get_current_user)):

→ auth.py:41-66  get_current_user()
  1. 从 Authorization 头提取 token
  2. jwt.decode(token, SECRET, [HS256])  ← 验证签名 + 过期时间
  3. user_id = int(payload["user_id"])   ← 从 payload 提取 ID
  4. db.query(User).filter(id=5).first() ← 查数据库确认用户仍存在
  5. 返回 User 对象

→ UserResponse.model_validate(user)
→ 返回 { id:5, username:"testuser", is_admin:false, ... }
```

**为什么 JWT 验证后还要查数据库**：如果管理员在此期间删除了该用户，token 仍然有效但用户不应再能访问。查数据库确保用户真实存在。这是防御性设计。

## 十三、异常路径

正常流程之外，每个节点都可能失败。以下是各阶段的异常处理：

### 13.1 前端阶段异常

| 异常 | 触发条件 | 处理 |
|------|----------|------|
| 重复提交 | 按钮快速点击两次 | `submitting` 守卫直接 `return` |
| 密码不一致 | `password !== confirmPassword` | 前端拦截，不发送请求 |
| 网络断开 | Axios 无 `response` 属性 | `e.response?.data?.detail` 可选链返回 undefined → 显示"注册失败" |
| Store 异常 | API 返回非预期格式 | catch 块显示通用错误信息 |

### 13.2 后端阶段异常

| 异常 | HTTP 状态码 | 触发条件 | 响应内容 |
|------|------------|----------|----------|
| 频率超限 | 429 | 同 IP 60秒内超过10次 | `{"detail":"请求过于频繁，请稍后再试"}` |
| 参数校验失败 | 422 | username<2字符 或 password<6字符 或 正则不匹配 | `{"detail":[{"loc":["body","username"],"msg":"..."}]}` |
| 用户名已存在 | 400 | 数据库查到同名用户 | `{"detail":"用户名已存在"}` |
| 数据库连接失败 | 500 | MySQL 不可用 | `{"detail":"服务器内部错误"}` (被全局异常处理器捕获并记录日志) |
| JWT 签发异常 | 500 | SECRET_KEY 配置异常 | `{"detail":"服务器内部错误"}` |

### 13.3 全局异常处理器

```python
# main.py:18-23
logger = logging.getLogger("bg3map")

@app.exception_handler(Exception)
async def global_exception_handler(request: Request, exc: Exception):
    logger.exception("未捕获的异常: %s %s", request.method, request.url.path)
    # 完整 traceback 写入日志，不暴露给客户端

```

**设计意图**：任何未被具体路由捕获的异常都会落到这里。日志记录完整 traceback 供排查，客户端只看到安全的 500 提示。这防止了 debug 模式下的堆栈泄露。

## 十四、安全纵深防御表

安全不应依赖单一防线。以下是从外到内的多层防御：

| 层 | 防线 | 文件 | 失败后果 | 绕过难度 |
|----|------|------|----------|----------|
| 1 | 前端防重复提交 | `RegisterView.vue` | 可能发起多个注册请求 | 低（禁 JS 可绕过） |
| 2 | IP 频率限制 | `rate_limit.py` | 可暴力尝试弱密码撞库 | 中（切换 IP 可绕过） |
| 3 | Pydantic 字段校验 | `schemas.py` | SQL 注入、XSS 可能落库 | 高（必须通过校验才到路由） |
| 4 | 用户名 UNIQUE 约束 | `models.py` | 重名用户 | —（数据库级硬约束） |
| 5 | bcrypt 哈希 | `auth.py` | 数据库泄露后明文密码暴露 | 极高（每密码需单独暴力破解） |
| 6 | JWT 签名 | `auth.py` | 伪造任意用户身份 | 极高（需要 SECRET_KEY） |
| 7 | 配置强制校验 | `config.py` | 弱密钥/无密钥运行 | —（启动时 crash 退出） |
| 8 | 全局异常捕获 | `main.py` | 堆栈信息泄露 | —（统一转 500） |
| 9 | 路径遍历防护 | `markers.py` | 删除服务器任意文件 | 高（resolve + 前缀校验） |
| 10 | 响应字段过滤 | `schemas.py` | password_hash 泄露 | —（Pydantic 自动排除） |

#### Axios 请求拦截器 — api/index.js

```
作用：在每个请求发出之前自动附加 Authorization 头

执行时机：每次 api.get/post/put/delete 之前

代码：
  const token = localStorage.getItem('token')

  }
  return config

"拦截器"是什么：
  Axios 把 HTTP 请求抽象为管道。请求拦截器是"前处理"——在请求发出前做点事。
  响应拦截器是"后处理"——在响应回来后、交给调用者前做点事。

为什么用 localStorage 而不是其他地方：
  localStorage 是浏览器提供的键值存储，数据永久保存直到用户清除。
  缺点：任何 JavaScript 都能读取（XSS 攻击可窃取 token）。
  Cookie 配合 httpOnly 更安全，但 SPA 场景下设置较麻烦。
```

#### Axios 响应拦截器 — api/index.js

```
作用：集中处理 401 状态码（token 过期/无效）

执行时机：每次 HTTP 响应回来之后

代码：
  if (err.response?.status === 401 && !isRedirecting) {
    isRedirecting = true
    localStorage.removeItem('token')
    window.location.href = '/login'
  }
  return Promise.reject(err)

  假设页面同时发起了 3 个请求（regions + categories + markers）
  且 token 全部过期。3 个请求同时返回 401。
  不用标志 → 3 次 window.location.href = '/login' → 浏览器历史记录混乱
  用了标志 → 只跳转 1 次

Promise.reject(err) 的作用：
  即使处理了 401，仍然把错误继续向上抛出
  让调用方的 catch 块也能知道发生了什么
```

#### `rate_limit_auth(request)` — rate_limit.py

```
作用：FastAPI 依赖注入函数，在注册/登录端点之前执行

依赖声明：
  @router.post("/register")
  def register(data, db, _=Depends(rate_limit_auth)):  ← 下划线接收返回值但不使用

执行：
  1. 读取 X-Forwarded-For 头 → 获取真实客户端 IP
  2. 如果没有代理头 → 读取 request.client.host
  3. key = "auth:127.0.0.1"
  4. 调用 RateLimiter.is_allowed(key)
     - 维护该 IP 的最近请求时间戳列表
     - 清理 60 秒前的时间戳
     - 如果当前列表 >= 10 → 返回 False
     - 否则追加当前时间戳 → 返回 True
  5. 返回 False → raise HTTPException(429)
  6. 返回 True → 放行

返回值：None（通过或抛异常），函数体正常结束即放行
```

#### `RateLimiter.__init__(max_requests, window_seconds)` — rate_limit.py

```
作用：创建一个限流器实例

参数：
  max_requests = 10     — 窗口内最大请求数
  window_seconds = 60   — 滑动窗口时长

内部状态：
  self._store = defaultdict(list)  — 字典：IP → [时间戳1, 时间戳2, ...]
  self._hits = 0                    — 全局计数器，累计所有请求数
```

#### `RateLimiter._clean(key, now)` — rate_limit.py

```
作用：清理某个 IP 的过期时间戳（滑动窗口核心逻辑）

执行：
  cutoff = now - 60  (当前时间减去窗口大小)
  self._store[key] = [t for t in self._store[key] if t > cutoff]
  只保留 cutoff 之后的时间戳 → 窗口外的自动丢弃
```

#### `RateLimiter.is_allowed(key)` — rate_limit.py

```
作用：判断某个 IP 的本次请求是否允许

执行：
  1. self._clean(key, now)          — 先清理过期
  2. len(self._store[key]) >= 10   — 还剩 ≥10 个 → False (超限)
  3. 不足 10 → 追加当前时间戳 → True (放行)

每 1000 次调用做一次全量清理（_full_clean）
  目的：清理已经完全没请求的 IP 的记录，防止内存无限增长
```

#### `get_db()` — database.py

```
作用：FastAPI 依赖注入函数，为每个 HTTP 请求提供独立的数据库会话

依赖声明：
  db: Session = Depends(get_db)

执行：
  db = SessionLocal()    — 从连接池获取一个连接，创建会话
  try:
      yield db            — 把 db 注入到路由函数
  finally:
      db.close()          — 请求结束后归还连接到连接池

为什么用 yield 而非 return：
  return → 函数结束 = 无法在请求结束后执行清理代码
  yield  → 函数暂停，注入 db，等路由函数执行完毕后继续执行 finally

"会话"是什么：
  不是 HTTP session / cookie 的那个 session
  是 SQLAlchemy 的 Session 对象 — 数据库事务的管理单元
  类似一个"购物车"，你可以 add/edit/delete 多个对象，最后 checkout (commit)
```

#### `get_current_user(token, db)` — auth.py

```
作用：从请求的 Authorization 头中解析 JWT，返回当前登录的 User 对象

依赖声明：

执行：
  1. oauth2_scheme 从请求头中提取 "Bearer eyJ..." 的 token 部分

  4. user_id = int(user_id_raw)             — 转为整数
  5. db.query(User).filter(id=user_id).first()  — 查数据库确认用户存在
  6. 返回 User ORM 对象

异常处理：

OAuth2PasswordBearer 是什么：
  FastAPI 的安全工具。它会解析请求头：

  → 传给 get_current_user 的 token 参数
```

#### `hash_password(password)` — auth.py

## 附录三：专有名词解释

按中英文对照，一句话说清"这个东西到底是干什么的"。

### 前端 (JavaScript / Vue 生态)

| 名词 | 一句话解释 |
|------|-----------|
| **Vue 3** | 一个 JavaScript 框架。你写 `.vue` 文件描述页面长什么样，它负责把数据渲染成 HTML。 |
| **Composition API** | Vue 3 的新写法。用 `ref()` 和 `computed()` 等函数组织逻辑，替代 Vue 2 的 `data()` / `methods` 对象写法。 |
| **`ref()`** | Vue 的"响应式变量"。给一个值套上 `ref()`，这个值变化时所有用到它的地方自动更新。类似 Excel 的公式联动——改一个格子，相关的格子自动算。 |
| **`computed()`** | 依赖其他 ref 的计算属性。只有依赖的值变了才会重新算，有缓存。比如 `isAdmin = computed(() => user.value?.is_admin)`。 |
| **`reactive()`** | 另一种响应式包装。ref 针对单个值，reactive 针对整个对象。但 Vue 3 社区更推荐用 ref。 |
| **`v-model`** | Vue 的双向绑定指令。`<input v-model="username">` 表示：输入框的值变化 → username 跟着变；username 变化 → 输入框也更新。 |
| **`v-if`** | Vue 的条件渲染指令。`v-if="show"` 为 false 时，这个元素不出现在 DOM 树中。 |
| **`v-for`** | Vue 的循环渲染指令。`v-for="m in markers" :key="m.id"` 为数组中每个元素生成一份 HTML。key 必须唯一，帮助 Vue 追踪哪些 DOM 要更新。 |
| **`@click`** | Vue 的事件绑定简写，等价于 `v-on:click`。`@click="doSomething"` 表示点击时执行 doSomething。 |
| **`@submit.prevent`** | 表单提交事件 + `.prevent` 修饰符阻止浏览器默认行为（刷新页面）。 |
| **Pinia** | Vue 的官方状态管理库。把多个组件共享的数据（如用户信息、标记列表）集中到一个地方管理，组件不再各自持有数据副本。 |
| **Pinia Store** | Pinia 中的一个"仓库"。本项目有两个：`auth.js`（用户/登录状态）和 `map.js`（地图/标记数据）。 |
| **Vue Router** | Vue 的官方路由库。把 URL 路径（如 `/login`）映射到对应的页面组件。 |
| **`router.push('/')`** | 编程式导航。在 JS 代码中控制页面跳转，等价于用户点击了指向 `/` 的链接。 |
| **`router.beforeEach`** | 全局路由守卫。每次跳转前都执行，用于检查登陆态：没登录就去 `/login`。 |
| **Axios** | 基于 Promise 的 HTTP 客户端。浏览器里用 `fetch()` 也能发请求，但 Axios 提供了拦截器、自动 JSON 解析、超时设置等更丰富的功能。 |
| **Axios 拦截器** | 在请求发出前/响应回来后插入自定义逻辑。本项目用请求拦截器注入 token，用响应拦截器处理 401。 |
| **`localStorage`** | 浏览器提供的本地键值存储。数据永久保存，清浏览器缓存才会丢。容量约 5MB。 |
| **`window.location.href`** | 直接改变浏览器地址栏 URL，触发整页刷新。和 router.push 的区别：router.push 只换 Vue 组件，不刷新页面；href 是"真的去那个地址"。 |
| **Vite** | 新一代前端构建工具。启动开发服务器极快（毫秒级），因为它用浏览器原生 ES Module 按需加载，不打包全部代码。 |
| **Vite Proxy** | Vite 内置的请求转发功能。前端在 `localhost:5173`，后端在 `localhost:8000`，浏览器禁跨域请求。Vite 在中间转发，浏览器以为自己一直在和 5173 通信。 |
| **`npm`** | Node.js 的包管理器。`npm install` 下载依赖，`npm run dev` 启动开发脚本。 |
| **SPA (Single Page Application)** | 单页面应用。整个网站只有一个 HTML 文件，所有页面切换由 JavaScript 在浏览器内完成，不再向服务器请求完整 HTML。 |
| **`async/await`** | JavaScript 的异步编程语法。`async` 函数返回 Promise，`await` 等待 Promise 完成并取结果。不用写回调嵌套。 |
| **Promise** | 一个"将来会完成或失败"的值的占位符。then/catch 分别在成功/失败时执行。 |
| **`try/catch/finally`** | 异常处理结构。try 块执行可能出错的代码，catch 捕获错误，finally 无论成败都执行（常用于解锁按钮）。 |
| **ES Module** | JavaScript 的模块系统。`import X from './x.js'` 引入，`export` 导出。浏览器和 Node.js 都支持。 |
| **可选链 `?.`** | `e.response?.data?.detail` 等价于 `if (e && e.response && e.response.data) return e.response.data.detail`。任何一层是 null/undefined 就短路返回 undefined。 |
| **Debounce (防抖)** | 高频事件只响应最后一次。用户在搜索框快速打字时，停止输入 300ms 后才发请求，避免每次按键都请求。 |
| **`$emit`** | Vue 组件通信方式。子组件 `$emit('close')` → 父组件 `<Popup @close="handler">` 接收。数据从子流向父。 |
| **Props** | Vue 组件通信方式。父组件 `<Popup :marker="data">` → 子组件 `props.marker` 读取。数据从父流向子。 |
| **Slot (插槽)** | Vue 组件的内容分发。`<Popup> <button>删除</button> </Popup>` 中的按钮会渲染到 Popup 模板中 `<slot>` 的位置。 |
| **`onMounted`** | Vue 生命周期钩子。组件第一次渲染到页面上后执行。常用于初始化数据加载。 |
| **`onBeforeUnmount`** | Vue 生命周期钩子。组件从页面移除前执行。常用于清理定时器、取消请求。 |
| **`nextTick`** | 等待 Vue 完成下一次 DOM 更新后再执行。比如先改了数据，等 Vue 重新渲染完成后，再去操作 DOM 元素。 |
| **`watch`** | Vue 的侦听器。当某个 ref 变化时自动执行回调。`watch(() => props.tileUrl, (url) => updateTile(url))`。 |

### 后端 (Python / FastAPI 生态)

| 名词 | 一句话解释 |
|------|-----------|
| **FastAPI** | Python 的异步 Web 框架。你写函数定义路由，它自动生成 OpenAPI 文档，支持类型校验和依赖注入。 |
| **ASGI** | 异步服务器网关接口。FastAPI 基于它运行在 Uvicorn 上，支持 WebSocket、长连接等异步特性。 |
| **Uvicorn** | ASGI 服务器。把 FastAPI 应用跑起来，监听 8000 端口等待 HTTP 请求。`uvicorn app.main:app --reload --port 8000`。 |
| **路由 (Route)** | URL 路径和处理函数的映射。`/api/auth/login` → 执行 `login()` 函数。 |
| **`@router.post("/register")`** | Python 装饰器语法。把下面定义的函数注册为 POST /api/auth/register 的处理器。 |
| **依赖注入 (Dependency Injection)** | 函数声明需要什么，框架自动提供。`def register(db=Depends(get_db))` → FastAPI 自动调用 `get_db()` 创建数据库会话，然后传给 `register`。不需要手动 new。 |
| **`Depends()`** | FastAPI 的依赖声明函数。告诉框架"这个参数的值请调用这个函数获取"。 |
| **Pydantic** | Python 的数据校验库。`class UserRegister(BaseModel)` 定义字段类型和校验规则。你传进来一个 dict，它自动校验并返回类型安全的对象。 |
| **`BaseModel`** | Pydantic 的基类。继承它就能使用自动校验、序列化、类型转换。 |
| **`field_validator`** | Pydantic 的自定义校验装饰器。在字段赋值后立即执行校验逻辑。 |
| **`model_validate()`** | Pydantic 的转换方法。把一个 dict 或 ORM 对象转为 Pydantic 模型实例并校验。 |
| **`model_dump()`** | Pydantic 的导出方法。把模型转为普通 Python dict（不含校验逻辑）。 |
| **`model_config = {"from_attributes": True}`** | 告诉 Pydantic 可以从 ORM 对象的属性（而不是 dict key）中读取数据。这是让 Pydantic 和 SQLAlchemy 协作的关键配置。 |
| **SQLAlchemy** | Python 最流行的 ORM 库。用 Python 类表示数据库表，用 Python 方法操作数据库，自动生成 SQL。 |
| **ORM (Object Relational Mapping)** | 对象关系映射。把数据库里的一行数据映射为 Python 中的一个对象。你操作 `user.name = "张三"` 而不是 `UPDATE users SET name='张三'`。 |
| **`Column(Integer, primary_key=True)`** | SQLAlchemy 的列定义。声明 `users` 表有一个叫 `id` 的整数主键列。 |
| **`relationship()`** | SQLAlchemy 的关联定义。声明 `users` 和 `markers` 是"一对多"关系——一个用户可以有多个标记。 |
| **`Session` (SQLAlchemy)** | 数据库会话。它不是 HTTP Session，而是"一次数据库对话"。你添加/修改/删除对象，最后 commit，整个过程是一个事务。 |
| **`db.add(user)`** | 把 User 对象注册到当前会话，标记为"待插入"。此时还没执行 SQL。 |
| **`db.commit()`** | 提交会话中的所有变更。SQLAlchemy 生成并执行 SQL，数据库写入。 |
| **`db.refresh(user)`** | 重新从数据库读取该行数据。用于获取数据库端生成的默认值（如自增 id、CURRENT_TIMESTAMP）。 |
| **`db.rollback()`** | 回滚。如果 commit 失败，撤销本次会话中的所有变更。本项目没有显式调用，SQLAlchemy 在异常时自动执行。 |
| **Alembic** | SQLAlchemy 官方的数据库迁移工具。你改模型 → 运行命令 → 生成迁移脚本 → 应用到数据库。像 Git 管理代码一样管理数据库结构变更。 |
| **JWT (JSON Web Token)** | 一个自包含的令牌格式。三部分：Header（算法声明）、Payload（用户 ID + 过期时间）、Signature（签名防篡改）。服务端签发后，客户端每次请求带着它，服务端验证签名即可确认身份。 |
| **bcrypt** | 专门设计用于密码哈希的加密算法。特点是"故意很慢"——每个哈希运算耗时约 0.1~0.3 秒。对正常用户无所谓，但对暴力破解是致命的（1 亿次尝试需要 1 年）。 |
| **Salt (盐)** | bcrypt 自动生成的一段随机数据，混在密码里一起哈希。确保相同密码产生不同哈希，防止彩虹表攻击。 |
| **彩虹表 (Rainbow Table)** | 预先算好的"明文→哈希"对照表。如果不用盐，攻击者可以直接查表找到密码。用了盐后，每个密码需要单独的彩虹表。 |
| **HS256 (HMAC-SHA256)** | JWT 的签名算法。HMAC = 用同一个密钥签名和验证。SHA256 = 哈希函数名。对称算法，签名和验证用同一个 key。 |
| **`python-jose`** | 处理 JWT 的 Python 库。提供 `jwt.encode()` 和 `jwt.decode()`。 |
| **`passlib`** | 处理密码哈希的 Python 库。提供统一的 API 操作多种哈希算法（bcrypt/argon2/...）。 |
| **CORS (Cross-Origin Resource Sharing)** | 浏览器的安全策略：`localhost:5173` 的 JavaScript 不能随便请求 `localhost:8000`，除非 8000 的响应头里说"我允许 5173 来访问"。 |
| **CORS Middleware** | FastAPI 中自动给每个响应添加 CORS 头的中间件。`allow_origins=[...]` 列出允许的源。 |
| **X-Forwarded-For** | HTTP 请求头。当请求经过代理（Nginx/Vite）时，代理把真实客户端 IP 写在这个头里传给后端。否则后端只能看到代理的 IP。 |
| **`Pathlib`** | Python 3.4+ 的路径处理库。`path / "subdir" / "file.txt"` 比 `os.path.join(path, "subdir", "file.txt")` 更直观安全。 |
| **`Path.resolve()`** | 将相对路径转为绝对路径，并消除 `..` 和 `.`。`"/a/b/../c"` → `"/a/c"`。 |
| **`os.getenv()`** | 读取系统环境变量。`os.getenv("DATABASE_URL")` 获取数据库连接字符串。 |
| **`dotenv`** | Python 库。在应用启动时把 `.env` 文件中的键值对加载为环境变量。这样敏感配置不用写在代码里。 |
| **`venv` (虚拟环境)** | Python 的隔离环境。每个项目有自己的 Python 包，安装的库不影响系统全局。`venv\Scripts\activate` 激活后，`pip install` 只安装到当前项目。 |
| **`requirements.txt`** | Python 项目的依赖清单。列出项目需要的所有包和版本。`pip install -r requirements.txt` 一键安装。 |
| **MySQL** | 关系型数据库。数据以表格形式存储，行=记录，列=字段。支持 SQL 查询、事务、索引。 |
| **`python-dotenv`** | 上面 `dotenv` 的 PyPI 包名。提供 `load_dotenv()` 函数。 |

### 数据库

| 名词 | 一句话解释 |
|------|-----------|
| **PRIMARY KEY (主键)** | 表中唯一标识每一行的列。`id INT PRIMARY KEY AUTO_INCREMENT` 表示自动编号，每插入一行自动 +1。 |
| **FOREIGN KEY (外键)** | 指向另一张表主键的列。`region_id INT REFERENCES regions(id)` 确保标记点的 region_id 必须对应一个真实存在的区域。 |
| **UNIQUE** | 唯一约束。`username VARCHAR(50) UNIQUE` 确保不能有两个同名用户。数据库层面硬约束。 |
| **AUTO_INCREMENT** | 自动递增。插入新行时不指定 id，MySQL 自动分配下一个可用编号。 |
| **INDEX (索引)** | 数据库的"目录"。给常用筛选列建索引，查询速度从全表扫描 O(n) 降到 B+树查找 O(log n)。 |
| **VARCHAR(255)** | 可变长字符串，最多 255 字符。 |
| **TEXT** | 长文本。适合描述内容，可以存几千字。 |
| **DECIMAL(10,2)** | 定点小数。总共 10 位，其中 2 位小数。比 FLOAT 精确，适合坐标和金额。 |
| **DATETIME** | 日期时间类型。存储 `2026-06-08 01:08:46`。 |
| **CURRENT_TIMESTAMP** | MySQL 函数，自动填入当前时间。 |
| **InnoDB** | MySQL 的默认存储引擎。支持事务、行级锁、外键约束。 |
| **事务 (Transaction)** | 一组 SQL 操作要么全部成功，要么全部撤销。转账：扣 A 的钱和加 B 的钱必须同时生效或同时不生效。 |
| **ACID** | 事务的四项保证：原子性(Atomicity)、一致性(Consistency)、隔离性(Isolation)、持久性(Durability)。 |

### 地图

| 名词 | 一句话解释 |
|------|-----------|
| **Leaflet.js** | 轻量级开源地图库。默认用于地理地图（经纬度），本项目通过 `CRS.Simple` 配置改用平面像素坐标。 |
| **CRS.Simple** | Leaflet 的坐标系配置。CRS = Coordinate Reference System。Simple 模式下坐标是平面 x/y 像素，不涉及经纬度投影。 |
| **瓦片地图 (Tile Map)** | 把一张大图切成 256×256 像素的小方块，只加载当前视口内的那些。Google Maps 的思路。 |
| **瓦片 URL 模板** | `"/TileMap/chapter1/地图名/{z}/{y}/{x}.png"`。Leaflet 根据当前位置和缩放级别自动替换 {z}/{y}/{x} 为实际值，按需加载瓦片。 |
| **zoom 级别** | 缩放等级。zoom=0 表示整个世界，每个 zoom+1 放大一倍。最大 zoom 由图像分辨率决定。 |
| **`marker` (标记点)** | 地图上的一个图钉/点。可以是传送点、怪物、道具等。有坐标 + 图标 + 点击信息。 |
